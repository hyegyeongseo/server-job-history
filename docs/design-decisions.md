# 설계 결정 기록 (Design Decisions)

이 프로젝트에서 "왜 그렇게 설계했는지"를 **대안·트레이드오프·한계**까지 남긴 문서.
각 항목은 `맥락(문제) → 고려한 대안 → 결정 → 트레이드오프/한계` 순서로 정리합니다.

> 요약은 [README의 설계 결정](../README.md#설계-결정-요약)에, 전체 맥락은 여기에.

---

## 0. 실행 환경 전제 (중요)

이 프로젝트는 **개인 노트북 1대 위 VMware Workstation VM 4대**(control-plane `cp-1` + worker `1~3`)에서 돌아갑니다.
클라우드 비용 없이 **쿠버네티스를 "쓰는" 수준이 아니라 "운영"까지** 경험하는 것이 목표였고,
그 대신 **자원(노트북 1대)·고가용성·규모**는 명확히 제약됩니다.

이 전제가 이후 모든 결정의 배경입니다. 특히 "노트북 자원을 아끼려 워커를 껐다 켜는" 상황이 실제로 발생했고,
그 과정에서 `local-path PVC 노드 고정`, `StatefulSet 파드가 죽은 노드에서 Terminating 정지` 같은 **현실 운영 문제를 직접 겪으며** 얻은 판단을 아래에 반영했습니다.

---

<a id="adr-001"></a>
## ADR-001 · 실행 환경 — 왜 관리형(EKS)이 아니라 kubeadm인가

- **맥락** — 컨트롤플레인·CNI·스토리지·인증서 같은 Kubernetes 내부 동작을 직접 이해하고, 관리형 서비스가 숨기는 운영 요소를 직접 경험하고 싶었습니다. 개인 프로젝트라 비용도 0에 가깝게 유지해야 했습니다.
- **대안**
  - (a) **EKS/GKE 등 관리형** — 편하지만 컨트롤플레인이 가려져 학습 목적에 안 맞고, 비용 발생.
  - (b) **minikube / kind** — 설치는 쉽지만 사실상 단일 노드라 **멀티노드 스케줄링·노드 장애·롤링** 경험이 얕습니다.
  - (c) **kubeadm on VM** — 직접 부트스트랩. 러닝커브는 크지만 전 과정을 손으로 다룹니다.
- **결정** — (c) **kubeadm on VMware VM 4대**. flannel(CNI) · MetalLB(LB) · local-path(스토리지) · metrics-server를 직접 구성.
- **트레이드오프 / 한계**
  - 컨트롤플레인 **HA 없음**(cp-1 단일). 운영이라면 3-노드 etcd/API가 필요.
  - 노트북 1대라 **모든 VM 상시 구동이 부담** → 워커를 껐다 켜며 작업하게 됨.
  - 그 제약이 오히려 **노드 다운 시 재스케줄, PVC 노드 고정** 같은 현실 문제를 체득하는 계기가 됨(→ [7. 스토리지](#7-스토리지--local-path-선택과-그-한계)).

<a id="adr-002"></a>
## ADR-002 · 배포 — 왜 GitOps(ArgoCD)인가

- **맥락** — 매니페스트를 `kubectl apply`로 직접 밀면 **클러스터 실제 상태와 git이 어긋나고**, 롤백·재현이 수동이 됩니다.
- **대안**
  - (a) **kubectl 수동 apply** — 가장 단순하지만 드리프트·재현성 문제.
  - (b) **CI(GitHub Actions)에서 kubectl push** — 자동화는 되지만 자격증명을 CI에 쥐여줘야 하고, 상태 동기화(self-heal)는 없음.
  - (c) **GitOps(ArgoCD) pull** — 클러스터가 git을 스스로 당겨 동기화.
- **결정** — (c) **ArgoCD**. git이 단일 소스, `selfHeal`로 드리프트 교정, **PreSync 훅**으로 DB 마이그레이션을 롤아웃 전에 멱등 실행.
- **트레이드오프 / 한계**
  - self-heal이 **HPA가 조정한 `replicas`를 매니페스트 값으로 되돌려** 플래핑을 유발 → `ignoreDifferences(/spec/replicas)`로 해결(실제 겪은 문제).
  - 컴포넌트·러닝커브 추가. 모니터링 스택은 아직 ArgoCD 밖(수동 helm)이라 **관리 범위가 반쪽** — 확장 여지로 남김.

<a id="adr-003"></a>
## ADR-003 · 관측성 — 왜 메트릭·로그·트레이스를 하나의 `trace_id`로 묶었나

- **맥락** — 메트릭만 보면 "느리다"는 알아도 **"왜 느린지"** 를 모른다. 로그·트레이스가 서로 따로 놀면 원인 추적에 시간이 든다.
- **대안**
  - (a) **메트릭만(Prometheus)** — 추세는 보이나 개별 요청의 원인 불가.
  - (b) **메트릭 + 로그** — 원인 로그는 있지만 요청 단위로 잇기 어렵습니다.
  - (c) **3신호 + 상관관계** — 메트릭·로그·트레이스를 공통 ID로 연결.
- **결정** — (c). django-prometheus(메트릭) + `python-json-logger` 구조화 로그 + OpenTelemetry(Django·psycopg2 계측) 트레이스. 모든 로그에 `trace_id`/`span_id`/`request_id` 주입 → Grafana에서 Loki 로그 ↔ Tempo 트레이스 교차 추적.
- **실제 효과** — 부하 중 **"요청 전체는 6초인데 DB 쿼리(SELECT)는 ms"** 인 상황을 트레이싱으로 확인해 **병목이 DB가 아님**을 규명(추측이 아니라 근거로).
- **트레이드오프 / 한계** — 계측 코드 + 수집 스택(Loki·Tempo·Alloy) 추가 부담. 트레이싱 오버헤드는 `OTEL_TRACING_ENABLED` 토글로 제어(Tempo 준비 후 켜는 순서도 관리).

<a id="adr-004"></a>
## ADR-004 · 비밀 관리 — 왜 sealed-secrets인가

- **맥락** — GitOps라 git이 소스인데, **Secret을 평문으로 git에 올릴 수 없습니다.** 그렇다고 클러스터에만 수동 주입하면 GitOps 재현성이 깨집니다.
- **대안**
  - (a) **평문 Secret 수동 주입** — 간단하지만 git 재현성 상실, 부트스트랩마다 수작업.
  - (b) **외부 Vault / KMS** — 강력하지만 **개인 lab에는 운영 부담이 과함**.
  - (c) **sealed-secrets** — 암호문(SealedSecret)만 git에 커밋, 컨트롤러가 클러스터 안에서 복호화.
- **결정** — (c). `kubeseal` 공개키로 봉인 → git 커밋 → 컨트롤러 개인키로만 복호화. **먼저 평문 Secret로 동작을 검증한 뒤 sealed로 전환**(점진적 도입).
- **트레이드오프 / 한계**
  - **봉인키(개인키) 분실 = 모든 SealedSecret 복호화 불가** → 주 1회 **백업 CronJob**(cp-1 디스크)으로 완화.
  - 단, 그 백업은 **같은 노트북 디스크**라 오프사이트가 아니고, **키 로테이션·외부 보관은 미구현**(한계로 명시). 운영이라면 오프사이트 + 로테이션 필요.

<a id="adr-005"></a>
## ADR-005 · 알림 — 왜 단순 임계가 아니라 SLO / error-budget인가

- **맥락** — "5xx가 N개 넘으면 알림"은 **순간 스파이크에 시끄럽고**, 사용자 체감(가용성)과도 어긋납니다.
- **대안**
  - (a) **고정 임계 알림** — 구현은 쉽지만 오경보/누락이 잦습니다.
  - (b) **단일 윈도우 SLO** — 낫지만 느리게 뜨거나 순간치에 흔들립니다.
  - (c) **멀티윈도우 멀티 burn-rate** — 빠른 감지 + 오경보 억제.
- **결정** — (c). 가용성 **99% SLO** + **5m·1h 동시 조건 fast-burn(14.4×)** 으로 **지속 장애만 page**. 기록룰(SLI)과 PrometheusRule로 구현.
- **실제 효과** — DB 장애를 주입하니 **가용성 SLI 100% → 41% 급락 + 5xx 상승 + burn 상승**이 대시보드에 실시간 반영됨(→ [README 결과](../README.md#결과--검증)).
- **트레이드오프 / 한계** — `99%`·`14.4×`는 **Google SRE 워크북의 권장 예시값**이지 절대 기준이 아닙니다 → 환경에 맞게 조정 대상([런북](runbooks/error-budget.md)에 근거·튜닝 표 명시).

<a id="adr-006"></a>
## ADR-006 · 도메인 — 왜 작업 삭제 금지 + 체인 + soft-delete인가

- **맥락** — 서버 운영 작업 이력은 **감사·추적의 대상**입니다. 지우거나 덮어쓰면 "누가 언제 무엇을" 했는지가 사라집니다.
- **대안**
  - (a) **일반 CRUD(삭제 허용)** — 단순하지만 이력이 훼손됩니다.
  - (b) **삭제 금지 + 정정 체인** — 이력을 보존하고 수정은 새 레코드로.
- **결정** — (b). Job **하드 삭제 API 비활성**, 수정은 **생성 후 30분 창**만 허용, 이후 정정은 **새 작업을 `previous_job`/`root_job`으로 체인 연결**. 서버는 **soft-delete**(복구는 ADMIN). 모든 생성/수정/삭제는 **감사 로그 + 변경 diff**를 트랜잭션으로 원자 기록.
- **트레이드오프 / 한계** — 데이터가 계속 누적되고, 조회 시 `is_deleted`/체인 필터링이 필요. 대신 **추적성·복구성**을 얻습니다.

<a id="adr-007"></a>
## ADR-007 · 스토리지 — local-path 선택과 그 한계

> Lab 환경의 제약이 가장 직접적으로 드러난 설계 결정.

- **맥락** — 온프렘 kubeadm lab에 분산 스토리지(Ceph/Longhorn)가 없지만, PVC는 필요합니다(postgres·loki).
- **대안**
  - (a) **hostPath** — 원시적, 관리 어려움.
  - (b) **local-path-provisioner** — 동적 프로비저닝, 설정 간단.
  - (c) **분산 스토리지(Longhorn 등)** — 복제·이동 가능하지만 자원·운영 부담 큼.
- **결정** — (b) **local-path**. lab 규모에 맞는 최소 복잡도.
- **트레이드오프 / 한계 (실제로 겪음)**
  - local-path PVC는 **프로비저닝된 노드에 고정**된다 → 그 노드를 끄면 해당 워크로드가 못 뜹니다. 실제로 **worker를 껐더니 postgres가 그 노드에 묶여 `Pending`**, prometheus/tempo는 죽은 노드에서 **`Terminating` 정지**(StatefulSet 특성)로 서비스가 끊겼습니다.
  - 즉 **단일 노드 장애 = 데이터 워크로드 중단**. 운영이라면 분산 스토리지 + 복제가 필요하다는 것을 체득. lab 규모라 이 한계를 수용하고, 노드 배치를 인지한 채 운영.

## 종합 — lab과 운영(production)의 간극

이 프로젝트는 "운영 패턴"을 **검증**하는 데 목적이 있고, 아래는 실제 운영으로 가려면 메워야 할 지점(인지하고 있음):

| 영역 | 현재(lab) | 운영이라면 |
|---|---|---|
| 컨트롤플레인 | 단일 cp-1 | 3-노드 HA(etcd/API) |
| 스토리지 | local-path(노드 고정) | 분산 스토리지 + 복제 |
| 비밀 백업 | cp-1 디스크 CronJob | 오프사이트 + 키 로테이션 |
| 모니터링 배포 | 수동 helm | GitOps로 편입 |
| CI/CD | 수동 빌드·push | 파이프라인 자동화(테스트·이미지·서명) |
| SLO 값 | 예시값(99%/14.4×) | 서비스 실측 기반 재설정 |

**핵심은** "무엇을 못 했나"가 아니라, **왜 그 선택을 했고 그 한계를 아는가**입니다.
lab의 제약 안에서 배포·관측·보안·장애 대응을 한 번씩 직접 겪어본 것이 이 프로젝트의 목적입니다.

## 배운 것 (Lessons Learned)

- **GitOps와 HPA는 기본 설정만으로 충돌**할 수 있습니다 — self-heal이 `replicas`를 되돌려 플래핑. `ignoreDifferences`로 소유권을 나눠야 한다는 걸 겪었다. ([ADR-002](#adr-002))
- **StatefulSet의 복구 전략은 스토리지 선택에 달렸습니다** — local-path는 PVC가 노드에 고정돼, 노드를 끄면 postgres가 `Pending`·prometheus가 `Terminating`으로 멈춘다(force-delete 필요). ([ADR-007](#adr-007))
- **trace_id 상관관계**가 있으면 메트릭만으로 못 찾는 병목을 짚을 수 있습니다 — "요청 6s인데 DB는 ms"를 트레이스로 규명. ([ADR-003](#adr-003))
- **Secret을 숨기는 것보다 봉인키를 잃지 않는 것이 더 중요**합니다 — 개인키 분실 = 전체 복호화 불가. 백업 자동화가 핵심. ([ADR-004](#adr-004))
- Lab 환경에서의 제약과 Production 환경에서 필요한 구성을 구분해 문서화하는 것이 설계 의사결정을 명확하게 전달하는 데 중요하다는 점을 배웠습니다.
